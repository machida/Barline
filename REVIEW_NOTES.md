# Preview Review Notes

プレビュー領域の楽譜記号・文字配置について、レビュー観点と修正案を集約するためのメモ。

## Current Findings

### 1. コード文字が衝突しやすい

- ディグリー表記、長いコードネーム、コード枠増加が重なると、コード同士が横方向で簡単に衝突する。
- 通常の4分割前提では読めても、6枠相当やディグリー併記では実用上かなり厳しい。
- 現状の根本原因は「枠を等分して左寄せ配置する」モデル。小節幅を広げても、1つの幅広コードが隣接スロットへ食い込みうる。
- Case B denseChords は、表示後コード幅ベースのスロット配分へ切り替えて解消済み。

修正案:

- 1小節の最小横幅をコード枠数だけでなく表示文字数でも増やす。
- ディグリー表記オン時は自動で1段あたりの小節数を減らす。
- コード位置を単純左寄せ固定ではなく、枠中央基準または簡易衝突回避で調整する。
- 長いコードやディグリー併記時はフォントサイズを1段階落とす。

### 2. 転調と拍子変更が同居すると小節頭が詰まる

- 同じ小節頭に調号変更と拍子変更が入ると、情報塊が大きくなり、直後のコード頭と近すぎる。
- 演奏者目線で、その小節の読み始め位置が少し曖昧になる。

修正案:

- 調号変更と拍子変更が同居する小節は専用の余白幅を増やす。
- 調号と拍子記号を少し縦方向に分節し、塊感を弱める。
- 最初のコード開始位置を強制的に右へ送る。
- 条件が重い段は自動で小節数を減らす。

### 3. Segno / Coda とコード・見出しが干渉しやすい

- Segno、Coda、継続セクションのインライン見出し、コード帯が同じ上部レーンに集まりやすい。
- 記号同士は避けられていても、全体として視線の優先順位が曖昧になる。

修正案:

- 上部要素を `見出し > 終括弧 > Segno/Coda > コード` のようにレーン分けする。
- Segno/Coda はコード位置計算から独立した固定アンカーで配置する。
- 継続セクション見出しがある小節では Segno/Coda を右へ逃がす。
- 上部が混む場合は、記号を五線上の専用マージン帯に逃がす。

### 4. `to Coda` がやや浮いて見える

- 右下寄せ配置のため、小節との紐付きが少し弱い。
- 段末や余白との関係によっては、飛び先指示より注釈に見えやすい。

修正案:

- 小節末尾のイベントとして見える位置に寄せる。
- `to` とコーダ記号の間隔を詰めて一体化して見せる。
- 必要ならガイド感のある配置にする。
- 段末では少し上げてページ余白に落ちないようにする。

### 5. 1カッコ・2カッコと長い任意カッコが競合する

- 標準の `1.` `2.` は問題が少ないが、長い任意カッコ文字列はコード帯に食い込みやすい。
- 文字量が増えた時の逃げ場が足りない。

修正案:

- 任意カッコ文字列が長い場合はラベルだけ縮小する。
- 一定文字数を超えたら省略表示し、編集値はそのまま保持する。
- カッコ帯とコード帯の縦間隔を広げる。
- 長い任意カッコがある段は自動で小節数を減らす。

### 6. 行頭の情報密度が高いケースで読み始めが重い

- 調号が多いキー、拍子変更、転調が重なると、ト音記号から最初のコードまでの情報密度が高い。
- 行頭の見通しが悪く、リードシートとしてはやや重たい。

修正案:

- 行頭ヘッダー幅を `調号数 + 拍子桁数 + 追加変更有無` に応じて拡張する。
- 調号が多いときはト音記号や拍子記号との間隔を広げる。
- 行頭に転調と拍子変更が重なる段は段全体を広めに再レイアウトする。
- 必要なら拍子記号サイズを少し下げる。

### 7. `Fine / D.C. / D.S.` が段末で窮屈になる

- 右下配置は自然だが、段末小節が狭いと終止線や反復記号に寄りすぎる。
- 指示記号としての視認性が下がる。

修正案:

- 段末小節では右下記号用の安全余白を確保する。
- 反復終止や終止線がある場合は少し左または下へ逃がす。
- 下部記号同士も固定間隔ではなく文字幅ベースで配置する。
- 幅不足時は段下の注記帯へ移す。

## Suggested Priority

1. コード文字衝突の抑制
2. 転調・拍子変更がある小節頭の余白整理
3. Segno / Coda / 見出し / カッコの上部レーン整理
4. `to Coda` と `Fine / D.C. / D.S.` の微調整

## Design Rules Before Fixing Everything

全項目に対応する場合、個別の場当たり対応を積み重ねるとレイアウト規則が崩れやすい。実装前に、以下の設計ルールを前提として揃える。

### 1. まず守るものを固定する

優先順位:

1. コードが重ならず読めること
2. 反復・反復終了・セクション頭など、進行を誤読させる記号が消えないこと
3. Segno / Coda / Fine / D.C. / D.S. などの導線記号が所属小節を見失わないこと
4. ページ内にできるだけ多く収めること
5. 装飾的な整列感や厳密な省スペース化

判断に迷った場合は、紙面効率より可読性を優先する。

### 2. レイアウトを3レーンで考える

各段は少なくとも次の3レーンに分けて考える:

- 上部レーン: セクション見出し、1カッコ/2カッコ、任意カッコ、Segno、Coda
- 五線レーン: ト音記号、調号、拍子、コード、反復記号、小節線
- 下部レーン: `to Coda`、`Fine`、`D.C.`、`D.S.`

個別記号ごとの微調整ではなく、「どのレーンに属するか」を先に固定してから位置調整する。

### 3. 幅不足時の処理順を固定する

横幅が足りないときは、次の順で逃がす:

1. その小節のコードフォントを少し下げる
2. 小節頭の追加情報ぶんの内部余白を増やす
3. 1段あたりの小節数を減らす
4. 長い補助文字列だけ省略表示する

逆に、重要記号を消す、コードを重ねる、別小節に見える位置へ押し出す、は最後まで避ける。

補足:

- フォント縮小には下限を設ける（例: 0.75 程度）。
- 下限でも `スロット幅 >= そのスロットの表示コード幅 + gap` を満たせない場合は、段の小節数を減らす。

### 3.5 コード枠内の配置モデルを明示する

- スロット幅は「等分した見た目」ではなく、「そのスロットの表示コード幅を収める」ことを優先する。
- 合格条件は `slotWidth >= renderedChordWidth + gap`。
- 幅見積もり、フォント縮小、段の小節数減少はこの式を満たすために連動させる。
- 空スロットは可読性優先で最小幅に縮むため、厳密な拍位置保持より「重ならず読めること」を優先する。

### 4. 高さ可変は最小限にする

- 段ごとに高さが大きく変わると全体の整列感が崩れる。
- 原則は「共通の段高 + 必要時のみ追加帯」で考える。
- 上部・下部記号のための余白は、記号のたびに個別拡張せず、帯としてまとめて扱う。

### 5. 小節ごとの局所回避より段全体の再配置を優先する

- 特定のコードだけ数pxずらす、特定の記号だけ少し下げる、という局所回避は積み重なると不揃いになりやすい。
- 条件が重い小節がある段は、その段全体の `barStart`、`barWidth`、収容小節数を再計算する方が規則性を保ちやすい。

### 6. 行頭は特別扱いする

行頭はト音記号、調号、拍子記号、転調、拍子変更が集中するため、通常小節と同じルールで押し込まない。

- 行頭ヘッダー幅は独立して見積もる
- 行頭に変更記号が重なる場合は、その段の収容小節数減少を許容する
- 調号なしの簡単なケースでは余白を取りすぎない

### 7. 所属感を失う逃がし方はしない

記号を見やすくするために位置を動かしても、「どの小節の指示か」が曖昧になる配置は避ける。

- `to Coda` は段下に逃がしても、該当小節末尾イベントに見えること
- `Fine / D.C. / D.S.` は段末でも終止線の装飾に見えないこと
- Segno / Coda は見出しと混ざらず、小節アンカーを保つこと

### 8. 省略対象と非省略対象を分ける

省略してよいもの:

- 長い任意カッコ文字列
- セクションメモ
- 補助的な説明テキスト

省略してはいけないもの:

- コード本体
- 反復系記号
- セクション識別子
- 導線記号そのもの

### 9. 実装順は「個別記号」ではなく「レイアウト基盤」から

推奨順:

1. 横幅見積もりルールの統一
2. 上部/五線/下部のレーン化
3. 段全体の再レイアウト条件整理
4. その後に各記号の個別調整

### 10. テストも同じ粒度で増やす

全対応を目指すなら、最低でも次のケース群が必要:

- 長いコード + ディグリー表記
- 転調 + 拍子変更の同居
- Segno/Coda + セクション継続見出し
- 長い任意カッコ
- 段末の `to Coda` / `Fine` / `D.C.` / `D.S.`
- 多調号キー + 2桁拍子 + ページまたぎ

未検証の仮説と、実際に崩れた再現ケースは分けて記録する。

## Acceptance Criteria

各修正は、主観だけでなく次の観点で「直った」と判断する。

### 共通

- 記号やコードが互いに重なっていない
- 「どの小節に属する指示か」が視覚的に追える
- 通常ケースの譜面で余白が不自然に増えない
- 既存の単純な4小節譜面が改悪されない
- 可能なものは主観判定ではなく、機械検証可能な条件へ落とす

### コード帯

- 長いコード、ディグリー併記、コード枠増加時でもコード同士が潰れない
- コード開始位置が小節線や転調/拍子変更表示と視覚的に競合しない
- フォント縮小が入っても、通常印刷距離で読める
- canonical cases に対して、隣接コードの `x + 推定幅` が次コードの `x` を超えない
- フォント縮小を入れる場合でも、定義した下限スケールを下回らない

### 上部レーン

- セクション見出し、1カッコ/2カッコ、任意カッコ、Segno、Coda が互いに誤読を生まない
- 継続セクション見出しと導線記号が混ざって見えない
- 長い任意カッコがコード帯へ食い込まない

### 下部レーン

- `to Coda`、`Fine`、`D.C.`、`D.S.` が段末でも終止線や反復記号に埋もれない
- 下部記号が隣小節や次段の要素に見えない

### 行頭

- 調号、拍子、転調、拍子変更が重なるケースでも、最初のコード位置が明瞭
- 調号の少ない簡単なケースでは左余白が過剰に広がらない

### ページ全体

- 1ページ内の段間隔に大きなムラがない
- ページまたぎ時に continued section や sectionHead の意味が崩れない

## Non-goals

今回の対応範囲から外すもの:

- 楽譜記号自体のデザイン刷新
- コード入力ルールや正規化仕様の変更
- JSONフォーマットの変更
- 編集UIの全面再設計
- フォント差し替え
- SVG から canvas 等への描画基盤変更

必要なら別タスクとして切り出す。

## Canonical Test Scores

回帰確認用に、最低限次の代表ケースを固定する。

### Case A: ベースライン

- 単純な4/4、4小節/段、転調なし、導線記号なし
- 目的: 通常譜面が改悪されないことの確認

### Case B: コード高密度

- 長いコード、6枠相当、ディグリー表記オン
- 目的: コード衝突と横幅見積もりの確認

### Case B2: コード高密度 + 移調 + ページまたぎ

- ディグリー表記 + 移調 + 複数ページ
- 目的: 表示コード幅が最大化した状態での回帰確認

### Case C: 小節頭高密度

- 転調 + 拍子変更 + 多調号キー + 2桁拍子
- 目的: 行頭と小節頭の読み始め確認

### Case D: 上部レーン競合

- 継続セクション見出し + 1カッコ/2カッコ + Segno/Coda
- 目的: 見出し、導線記号、括弧の優先順位確認

### Case E: 下部レーン競合

- 段末に `to Coda`、`Fine`、`D.C.`、`D.S.`、反復終止
- 目的: 段末指示記号の所属感と余白確認

### Case F: ページまたぎ

- 複数ページ、continued section、sectionHead あり
- 目的: ページ分割後の意味保持確認

### Case G: ほぼ空白ページ

- 短い曲、導線記号なし、行数少なめ
- 目的: 段高調整後の間延び回帰確認

### Case H: 長い任意カッコ / 長いセクションメモ

- 長い endingText、長い section note
- 目的: 省略表示と上部帯の競合回帰確認

可能なら各ケースは固定JSONまたは生成用データとして保存する。

## Dependencies

着手順を誤ると、後の修正でやり直しが増える。依存関係は次の通り。

- 横幅見積もりの統一は、コード衝突対策と行頭整理の前提
- 上部/五線/下部のレーン定義は、Segno/Coda、任意カッコ、下部記号調整の前提
- 段高とページ容量の調整は、上部/下部レーンの整理後に再計算が必要
- `continued` / `sectionHead` 周辺の見出し整列は、上部レーン整理と密接に連動
- `to Coda`、`Fine`、`D.C.`、`D.S.` の微調整は、下部レーン基準確定後に行う

## Risk Notes

全対応で壊れやすい箇所:

- ページネーション
- `continued` セクションの結合表示
- `sectionHead` による頭位置移動
- `repeatStart` と double bar line の優先関係
- 調号なし/単純拍子ケースでの過剰余白
- 印刷時と画面時の見え方の差
- フォント読み込み差による文字幅ずれ

## Validation Method

確認手段を混ぜず、最低限次を通す:

1. Nodeテストまたは文字列ベース検証でレイアウト分岐を確認
2. 実ブラウザでスクリーン表示を確認
3. A4相当の印刷プレビューまたはPDF相当出力で確認

必要に応じて、単発の観察メモではなく「どのケースで何を確認したか」を追記する。

補足:

- 印刷相当確認が形骸化しないよう、スクリーンショット生成ハーネスを `tests/` 側へ寄せる余地あり。
